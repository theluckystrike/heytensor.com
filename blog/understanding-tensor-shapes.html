Understanding Tensor Shapes: The #1 ML Debugging Problem

Published January 2025 · 12 min read

If you have spent any time building neural networks, you know the feeling. You define your model, prepare your data, call model(input), and get hit with a cryptic error about mismatched dimensions. You stare at the shapes, count on your fingers, and eventually find the off-by-one error in your convolution padding. This article provides a systematic approach to never getting stuck on shape errors again.

Why Shapes Are Hard

Tensor shapes are hard for a specific reason: each operation transforms dimensions differently, and the rules are not intuitive. A 2D convolution reduces spatial dimensions based on kernel size, stride, and padding. A linear layer only touches the last dimension. Reshape can change the number of dimensions entirely. Keeping track of how these compose through 10, 20, or 50 layers is error-prone mental arithmetic.

The problem is compounded by batch dimensions (which most operations ignore), channel dimensions (which Conv2d changes but MaxPool does not), and the different conventions between PyTorch (NCHW) and TensorFlow (NHWC).

The Shape Propagation Mental Model

Think of your model as a pipeline of shape transformations. Each layer takes an input shape and produces an output shape according to a formula. Your job is to ensure that every layer's expected input shape matches the previous layer's output shape.

Input: [batch, 3, 224, 224]
  |
  v Conv2d(3, 64, kernel=7, stride=2, padding=3)
  |   Formula: H_out = floor((224 + 2*3 - 7) / 2) + 1 = 112
  |
  v [batch, 64, 112, 112]
  |
  v MaxPool2d(kernel=3, stride=2, padding=1)
  |   Formula: H_out = floor((112 + 2*1 - 3) / 2) + 1 = 56
  |
  v [batch, 64, 56, 56]
  |
  v Flatten(start_dim=1)
  |   64 * 56 * 56 = 200704
  |
  v [batch, 200704]
  |
  v Linear(200704, 1000)
  |
  v [batch, 1000]

The Conv2d Formula

The 2D convolution output formula is the most commonly needed:

H_out = floor((H_in + 2 * padding - kernel_size) / stride) + 1
W_out = floor((W_in + 2 * padding - kernel_size) / stride) + 1

Some common patterns to memorize:

  • kernel=3, stride=1, padding=1: Output size equals input size (most common in ResNets)
  • kernel=3, stride=2, padding=1: Output size is roughly half the input (downsampling)
  • kernel=1, stride=1, padding=0: Only changes the channel dimension (1x1 convolution)
  • kernel=7, stride=2, padding=3: Output is roughly half the input (ResNet stem)

The Flatten Trap

The most common shape error in practice happens at the transition from convolutional layers to fully connected layers. After your last pooling layer, you need to flatten the spatial dimensions into a single vector. The size of that vector depends on every preceding layer's parameters.

# Wrong: hardcoded value that does not match the actual shape
self.fc1 = nn.Linear(512, 256)  # But the actual flattened size is 2048!

# Right: calculate or use adaptive pooling
# Option 1: Calculate manually
# After conv layers: [batch, 128, 4, 4] -> flatten -> [batch, 2048]
self.fc1 = nn.Linear(128 * 4 * 4, 256)

# Option 2: Use adaptive average pooling to get a fixed size
self.pool = nn.AdaptiveAvgPool2d((1, 1))  # Always outputs [batch, C, 1, 1]
self.fc1 = nn.Linear(128, 256)  # C channels, always works

Adaptive average pooling is the robust solution. ResNets use it to decouple the classifier from the spatial resolution of the input.

Batch Dimension Rules

In PyTorch, the batch dimension is always dimension 0. Most operations preserve it silently:

  • Conv2d, MaxPool2d, BatchNorm: Operate on C, H, W dimensions (dims 1, 2, 3). Batch is untouched.
  • Linear: Operates on the last dimension only. Works with any number of batch dimensions.
  • Flatten(start_dim=1): Flattens everything except the batch dimension.
  • Reshape: You must account for the batch dimension explicitly.

When using torch.reshape or view, always use -1 for the batch dimension to avoid hardcoding batch sizes:

# Wrong: hardcodes batch size
x = x.view(32, -1)

# Right: uses -1 for batch dimension
x = x.view(-1, 128)
# or equivalently
x = x.flatten(1)

Debugging Strategy

When you hit a shape error, use this systematic approach:

  1. Print shapes at every layer. Add print(x.shape) in your forward method before each operation. This immediately shows where the mismatch happens.
  2. Work backwards from the error. The error message tells you what shape the operation received and what it expected. Trace backwards to find which layer produced the wrong shape.
  3. Use the formula. For Conv2d and MaxPool2d, plug numbers into the output formula. For Linear, check that the last dimension of the input matches in_features.
  4. Use a shape calculator. Tools like HeyTensor let you experiment with layer parameters and see the effect on shapes instantly, without running any code.
class DebugModel(nn.Module):
    def forward(self, x):
        print(f"Input: {x.shape}")
        x = self.conv1(x)
        print(f"After conv1: {x.shape}")
        x = self.pool(x)
        print(f"After pool: {x.shape}")
        x = x.flatten(1)
        print(f"After flatten: {x.shape}")
        x = self.fc1(x)
        print(f"After fc1: {x.shape}")
        return x

Common Architecture Shape Patterns

Understanding standard architecture patterns helps you reason about shapes faster:

ResNet Block

Uses kernel=3, padding=1 throughout so spatial dimensions stay constant. Skip connections require matching shapes, which is why ResNet uses 1x1 convolutions to adjust channel dimensions when needed.

Transformer

Input shape is [batch, seq_len, d_model]. Multi-head attention reshapes to [batch, num_heads, seq_len, d_head] where d_head = d_model / num_heads. The key shape transformation is this reshape-transpose-reshape cycle. AI workflow tools like ClaudKit can help manage the complexity of building transformer architectures.

U-Net

Encoder halves spatial dimensions (stride=2 conv or pooling). Decoder doubles them (transpose convolution or interpolation + conv). Skip connections concatenate along the channel dimension, doubling it.

Shape errors are not a sign of incompetence — they are the natural result of working with high-dimensional data and operations with non-obvious transformation rules.
